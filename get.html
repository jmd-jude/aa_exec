Obi: A Human-Centered AI Companion

Transforming Government Services Through Intelligent Conversation

Executive Summary

Strategic Overview

GET has established a mature market presence in digital identity systems and government infrastructure, yet citizens continue to struggle with complex processes. Common services like passport renewal—where one in five appointments fail due to documentation errors—remain confusing for users. Obi represents an opportunity to bridge this gap by integrating an AI customer experience layer with GET's core digital identity offerings, transforming frustrating government interactions into guided, accessible experiences.

Understanding the AI Implementation Challenge

While modern AI systems create the impression of having continuous memory and awareness, they are fundamentally stateless—meaning they don't inherently "remember" previous interactions or maintain an understanding of where users are in multi-step processes. Creating the reliable, guided experience that government services require demands a carefully engineered architecture with three critical components:

  • State Management: A structured system to maintain and update essential facts about the user throughout their journey (e.g., passport condition, name change status, documentation availability)
  • Journey Orchestration: Logic to track and manage where users are in multi-step processes, ensuring they receive appropriate guidance at each stage
  • Context Enrichment: Carefully crafted supplemental information that maximizes the AI's ability to provide reliable, accurate responses for specific government service scenarios

This architectural approach is what enables the seemingly conversational, knowledgeable experience users expect, while addressing the underlying technical constraints of AI systems.

Solution Approach

Our assessment indicates that with these three components properly implemented, AI can effectively enhance GET's existing systems while achieving the reliability threshold required for government services. We've validated this approach using passport renewal as a strategic test case because it features:

  • Deterministic Policy Logic: Clear, published guidelines with conditional requirements
  • Structured Document Requirements: Standardized validation criteria similar to other GET identity services
  • Discrete Path Branching: Well-defined user journeys that can become templates for other services
  • Clear Success Metrics: Objectively measurable outcomes for technical performance and business impact

Recommended Path Forward

We recommend developing a Proof of Concept to demonstrate how Obi can enhance GET's existing services while validating two critical hypotheses:

The POC will deliver:

  • Architecture documentation showing how Obi integrates with GET's existing identity verification systems
  • A model for estimating implementation effort across additional government services

Implementation Approach: How Obi Thinks

Creating Stateful Conversations with Stateless Technology

Large language models (LLMs) are inherently stateless—processing each message without retaining memory of previous interactions. Yet government services require continuous, reliable interactions that track user details and follow complex policy rules.

Our implementation addresses this fundamental challenge through three critical components that together create the illusion of memory and reasoning:

1. State Management

Structured flags that track key user information across conversation turns

2. Journey Orchestration

Logic that determines what phase of the process users are in and what comes next

3. Context Enrichment

Policy guidelines and conversation history provided to the LLM at each turn

This architecture transforms one-off AI interactions into coherent conversations where the system reliably "remembers" critical details and guides users through complex government requirements.

State Management: The Core Innovation

The success of Obi depends on maintaining a precise state object that tracks what we know about the user. This state isn't just a memory tool—it's the foundation for correct policy decisions and personalized guidance.

State Flags

{ "passport_condition": "damaged", "name_change": true, "has_name_change_doc": true, "passport_lost": false, "issued_within_15_years": true, "travel_within_6_weeks": true }

Policy Logic

if (damaged || lost || >15yrs || under16 || name_change w/o doc || traveling soon) return DS-11 else return DS-82

When a user like Sarah mentions her passport is damaged, the system:

  1. Updates her state flags (passport_condition: "damaged")
  2. Applies policy logic (damage triggers DS-11 requirement)
  3. Determines the next appropriate step in her journey (Phase 2: explanation)
  4. Enriches the LLM prompt with this context and policy requirements
  5. Generates a response that acknowledges her situation and guides her forward

This happens seamlessly between conversation turns, creating the illusion of a system that "remembers" everything about her situation.

A Scalable Framework for Government Services

While passport renewal serves as our initial test case, this architecture provides a template for any government service:

  • Define the state flags relevant to each service (eligibility criteria, document requirements)
  • Map the policy logic that determines service paths
  • Create journey flows that guide users through the process
  • Design context enrichment to maximize LLM accuracy

Our POC will validate this approach and provide a model for estimating implementation effort across multiple services. This creates a repeatable framework that can transform citizen experiences across GET's government identity portfolio.
